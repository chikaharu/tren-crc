## 1. 背景と問い

`tren-crc` の `Frame32` (32 × `u32` = 1024 bit) は対角 32 ビット (slot `i` の bit `i`) すべてを CRC-32C のシンドロームに割り当てている。CRC-32C 単体では、ランダム n ビット誤りに対して "n がハミング距離 (HD) を超えた領域" で漸近的に `1 − 2^−32 ≈ 99.999 999 98 %` の検出率となる。

「対角 32 ビットの一部を **補助パリティ層** に振り替えれば、特定の誤りクラス (とくに偶数ビット誤り) に対して検出率を改善できないか」が本タスクの問いである。トレードオフは明らか:

- CRC のシンドローム空間が `2^32` から `2^(32−k)` に縮む → 任意の "ランダムな大規模誤り" の検出率は `1 − 2^−(32−k)` に劣化する
- 補助 k ビットは CRC とは独立な性質 (例: 行内の偶数ビット反転は CRC が見落としても行 XOR は変化する、など) を測れる *かもしれない*

掃引する layout:

| layout | 補助ビット k | 補助の中身 | CRC 残量 |
|--------|-------------|-----------|---------|
| `Crc32Only` (production) | 0 | — | CRC-32C 32 bit |
| `RowXorPlusCrc { k }` | 1, 2, 4, 8, 12, 16 | 行 XOR (slot `i` の body 偶奇) | CRC-32C 下位 32−k bit |
| `ColXorPlusCrc { k }` | 1, 2, 4, 8, 12, 16 | 列 XOR (bit position `c` を全 slot で XOR) | CRC-32C 下位 32−k bit |
| `SplitOddEven16` | 16 (偶) + 16 (奇) | **対角偶数位置** (bit 2j of slot 2j) = body 全 1024 bit の **CRC-16/IBM** (poly `0x8005`、(x+1) 因子あり) / **対角奇数位置** (bit 2j+1 of slot 2j+1) = body の "偶数ビット位置だけ取り出した 512 bit" の **CRC-16/CCITT-FALSE** (poly `0x1021`) | 0 |

## 2. 理論セクション

### 2.1 CRC-32C 多項式の性質

CRC-32C (Castagnoli) の生成多項式 G(x) は

```
G(x) = x^32 + x^28 + x^27 + x^26 + x^25 + x^23 + x^22 + x^20
     + x^19 + x^18 + x^14 + x^13 + x^11 + x^10 + x^9 + x^8 + x^6 + 1
```

すなわち polynomial value `0x1EDC6F41` (x^32 を暗黙の最上位として除く 32 ビット表現)。係数の総数 (1 の個数) は 18 → `G(1) = 18 mod 2 = 0`、つまり **(x + 1) が因子**。

#### (x + 1) 因子から導かれる性質

- 任意の **奇数重み (奇数ビット反転)** 誤りは、誤り多項式 e(x) が `e(1) = 1` → (x + 1) で割り切れない → `G(x) | e(x)` は不可能 → **すべての奇数誤りは確定的に検出される (検出率 100 %)**
- 偶数重み誤りは `e(1) = 0` で (x + 1) を因子に持つ。残りの因子 H(x) (deg = 31) で割り切れる場合のみ CRC が見落とす → **偶数誤りの "ランダム miss 率" は理論上 `2^−31`** (`2^−32` ではない)

#### CRC-32C のハミング距離 (HD)

CRC-32C は HD = 6 を block length 16,360 bit までで保証 (Koopman 2002 の表より)。1024 + 32 = 1056 bit のフレーム長は十分この範囲内。したがって:

- **n ≤ 5 ビットの任意誤りは確定的に検出される**
- n = 6, 7, ... の偶数・奇数誤りに対しては、奇数なら 100 %、偶数なら少数の alias パターンを除き高確率で検出
- 32 ビットまでの **連続バーストはすべて検出される** (CRC-N の古典的性質)

### 2.2 行 / 列 XOR 補助層の理論的予測

`k` ビットを CRC から奪って行 XOR / 列 XOR に振り替えると:

| 観点 | 変化 |
|------|------|
| ランダム任意誤り miss 率 | `2^−32` → `2^−(32−k)` (劣化) |
| 単発ビット反転 (HD ≥ 2) | 100 % (不変) |
| ≤ 5 ビット任意誤り | 100 % (HD = 6 が 32−k ビット CRC でも維持される範囲では) |
| 行 XOR が "捕まえる" 誤り | スロット `i ∈ 0..k` の行内で奇数ビット反転 → aux パリティ変化を確定検出 |
| 列 XOR が "捕まえる" 誤り | bit position `c ∈ 0..k` の列内で奇数ビット反転 → aux パリティ変化を確定検出 |

行 XOR / 列 XOR は線形写像であり、CRC-32C も線形写像。両方とも GF(2) 線形なので、補助パリティが「CRC が見落とす誤りパターン」を独立にカバーする保証はない。多くの場合、aux パリティが捕まえる誤りクラスは CRC も既に捕まえているクラスと重複する。

### 2.3 `SplitOddEven16` の構成と理論

対角 32 位置を偶奇で分割:

- **対角偶数位置** (16 bit) = bit `2j` of slot `2j` for `j ∈ 0..16`
- **対角奇数位置** (16 bit) = bit `2j+1` of slot `2j+1` for `j ∈ 0..16`

各 16 ビットに置く値:

#### 対角偶数位置 16 bit = CRC-16/IBM(全 1024 bit body)

- Polynomial: `0x8005` (= x^16 + x^15 + x^2 + 1, 4 項 → 偶数 → **(x + 1) 因子あり**)
- Init: `0x0000`, refin/refout=true (reflected poly `0xA001`)
- 入力: 32 slot × 4 byte = 128 byte (各 slot は対角クリア後の `u32` を little-endian)
- 性質: (x + 1) 因子により **body 内の全奇数誤りを確定検出**。HD = 4 を block length 32,751 bit までで保証 (Koopman 2002)
- 期待効果: 「奇数及び全体誤り訂正」(ユーザ仕様) の "全体" 部分を担う

#### 対角奇数位置 16 bit = CRC-16/CCITT-FALSE(body の偶数ビット位置 512 bit)

- Polynomial: `0x1021` (= x^16 + x^12 + x^5 + 1, 4 項 → 偶数 → **(x + 1) 因子あり**)
- Init: `0xFFFF`, refin/refout=false
- 入力: 32 slot から各 slot の偶数ビット位置 (b ∈ {0, 2, ..., 30}) のみ抽出した 16 bit × 32 = 512 bit = 64 byte
- 性質: (x + 1) 因子により **入力 (= 偶数ビット位置投影) 内の全奇数誤りを確定検出**
- 期待効果: ユーザ提案の「偶数誤り訂正特化」の核心。多くの偶数重み body 誤りは "偶数ビット位置投影" 上では奇数重み誤り (= 偶数位置と奇数位置に分散している場合) になり、(x + 1) 因子で確定検出される。(両方とも偶数位置にある場合のみ投影上も偶数重みになり、これは CRC-16/CCITT 本体の HD = 4 性質に頼る)

#### 構成上の注意点

- 多項式 `0x1021` の "(x+1) 因子なし" は **誤った仕様** (4 項 → 偶数項数 → (x+1) は必ず因子)。本実装ではユーザ提案を素直に採用し、両方とも (x+1) 因子を持つ構成とする。"specialization for even errors" の効果は **多項式自体ではなく "偶数ビット位置投影" の入力前処理** から来る
- 16 + 16 = 32 bit の総シンドローム量は CRC-32C と同じ。差は "32 ビット連続シンドローム" vs "2 つの独立 16 ビットシンドローム" の構造のみ

### 2.4 期待される結果

本実験のスケール (n ≤ 32 ビット, 連続バースト ≤ 32 ビット) では、Crc32Only / RowXorPlusCrc / ColXorPlusCrc / SplitOddEven16 いずれも HD ≥ 4 と CRC のバースト検出特性によって **すべて 100 % 検出** に張り付くはずである。差が出るとすれば n ≫ 32 の領域。本タスクのスペック内では否定的結論 (= 改善なし) が予想される。

## 4. 考察

### 4.1 結論: 否定的 (negative result)

スペック内のすべての (layout, k, error_class, n) 組み合わせで検出率は 10000 / 10000 となり、現スペック範囲では補助パリティ層を入れても CRC-32C 単体に対する **観測可能な改善は得られなかった**。同時に、CRC を 32 → 32−k ビットに削っても **観測可能な劣化も出なかった**。`SplitOddEven16` (ユーザ提案の対角偶奇分割) も他の 3 layout と同じ結果。

### 4.2 なぜそうなるのか

- **HD ≥ 4 が支配的**: テストした最小誤り規模 (n = 1) から最大 (n = 32 + バースト 32) まで、CRC-32C の HD = 6, CRC-16/IBM・CRC-16/CCITT の HD = 4 と 16 ビットバースト保証によって理論上 100 % 検出が予測される領域である。`2^−(32−k)` の "ランダム miss" は試行回数 `10^4` × 最大 `k = 16` で期待値 `10^4 × 2^−16 ≈ 0.15 ミス` であり、観測不能
- **奇数誤りは (x + 1) 因子で確定検出**: 4 layout 全部 (CRC-32C, CRC-16/IBM, CRC-16/CCITT すべて (x+1) 因子あり) が odd 行 100 % は当然
- **行 XOR / 列 XOR / 偶奇分割いずれも線形写像**: GF(2) 線形同士で誤りパターンが独立にカバーされる保証はなく、CRC が見落とすパターンは補助でも見落とされやすい
- **`SplitOddEven16` の "偶数誤り特化" は理論上意味があるが、この試行規模では計測不能**: 期待効果は body 内偶数重み誤りのうち "偶数ビット位置だけに集中している" ものの検出率を上げること。本実験の偶数誤り (n = 2, 4, ..., 32) は body 1024 bit からの一様サンプリングなので、偶数位置 512 bit のみに全部入る確率は (1/2)^n × C(n, n) など極小であり、差が出る前に CRC-32C 自身の HD = 6 で全部捕まえる

### 4.3 トレードオフ曲線について

理論上のトレードオフは

```
任意誤り miss 率 ≈ 2^−(32−k)     (劣化, k に対し指数的)
偶数誤り改善幅   ≈ ε(k, layout)   (実測上 ≤ Wilson lower bound 1 − 0.999616 ≈ 3.84 × 10^−4)
```

`ε(layout) > 2^−(32−k) − 2^−32` を満たす layout が存在すれば純改善があるが、本実験では `ε ≈ 0` (差が観測されない) であり、左辺が右辺を上回る組み合わせは存在しなかった。

### 4.4 何を確定できたか / できなかったか

- ✅ **本タスクのスペック範囲 (n ≤ 32, burst ≤ 32) では補助パリティ層は無意味**。`Crc32Only` (production, k = 0) を維持すべき
- ✅ **行 XOR / 列 XOR / 偶奇分割 CRC は GF(2) 線形である限り CRC と独立な情報源にはなりにくい**、という代数的直観の実験的裏付け
- ✅ **`SplitOddEven16` の round-trip + 単発ビット反転 検出はユニットテストで担保済み** (`flip_one_body_bit_breaks_split_verify`)
- ❌ **n ≫ 32 領域での挙動は未確認**。`2^−(32−k)` が顕在化する k = 16 あたりで n = 64..256 ビット程度の誤りを 10^6..10^7 trial 回せば差は出るはず。`SplitOddEven16` も同様に n ≫ 32 で初めて CRC-32C との差が出るはず。"production CRC を縮めて aux に振り替えるな" の結論は変わらないが、aux パリティが純粋に害となる k の閾値を実測で出すことはできる (**フォローアップ候補**)
- ❌ **誤り訂正 (ECC) としての挙動は本実験のスコープ外**。本実験は検出率のみ測っており、`SplitOddEven16` の 2 つの 16 ビットシンドロームを使った訂正能力 (例: 1 ビット誤りの位置特定) は別途評価すべき (**フォローアップ候補**)
- ❌ **CRC の代わりに非線形関数 (例: SipHash 32-bit truncation) を入れた場合の挙動**は本実験のスコープ外

### 4.5 実験の意義

本タスクは設計判断として「CRC-32C 32 ビット全部使う以外のレイアウトを検討すべきか?」に対する **明確な "No"** を残した。今後 `Frame32` の対角レイアウト変更提案 (例: aux パリティ追加 / scatter pattern 変更 / 偶奇分割) が出た場合、本ドキュメントを根拠に「検出率改善目的では却下」と言える。

**補助パリティ層 / 偶奇分割に意味があり得るのは検出率改善ではなく、(a) 訂正 (ECC), (b) 部分復号 (frame body の早期 reject), (c) 並列計算による latency 改善のいずれかの目的であり、それは別タスクで評価すべき**。
