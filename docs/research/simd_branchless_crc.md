## 1. 背景と問い

production の `Frame32::update_crc` / `verify_crc` は `crc32c` クレート経由で SSE 4.2 `_mm_crc32_*` 命令 (x86) または ARMv8 CRC 命令を使っている。命令がない CPU ではソフトウェアテーブルにフォールバックする。

本タスクは **CRC-32C 演算子を「分岐なし論理ゲート (XOR / AND / OR / NOT / NAND / XNOR / popcount)」のみで合成する複数の実装を書き、ベンチマークで比較する** ことで以下を定量化する:

1. **副作用ゼロ・分岐ゼロ実装の性能上限** (将来の暗号文脈サイドチャネル耐性下地)
2. **SIMD レジスタ (SSE2 / AVX2) で複数フレームを並列処理した場合のスループット**
3. **CRC が GF(2) 線形写像であることを最大限活用した "popcount-XOR" 実装**
4. ハードウェア CRC 命令との性能差を ns/frame と GiB/s で定量化

期待される結果は「ハード命令には敵わない」だが、否定的結論でも価値がある。現代 CPU の SIMD/popcount 性能上限を CRC-32C 文脈で測ったベンチが残ること自体が、将来の代替実装提案 (例: 暗号目的、移植性目的) を判断する材料となる。

実装一覧 (すべて `experimental` feature 配下、`src/research/crc_impls.rs`):

| 関数 | アプローチ |
|------|----------|
| `crc32c_hw` | `crc32c` クレート (黄金値, SSE 4.2 `_mm_crc32_u32`) |
| `crc32c_table` | Sarwate 8-bit single table (256 entries) |
| `crc32c_slice_by_8` | Intel slice-by-8 (8 tables × 256 entries) |
| `crc32c_branchless_bitwise` | bit-by-bit, 分岐ゼロ (`mask = (state & 1).wrapping_neg()`) |
| `crc32c_branchless_bitwise_v2` | 同アルゴリズム、`bool_ops::xnor/nand/xor3` 経由 |
| `crc32c_popcount_xor` | 線形性活用: `crc(M) = c_L ^ XOR_{set bits i} G_L[i]` |
| `crc32c_simd_sse2_x4` | SSE2 4 レーンに 4 メッセージの bit-by-bit を packing |
| `crc32c_simd_avx2_x4` | 同上を AVX2 (256-bit) 上で実行 (上位 4 レーン未使用) |

## 2. 理論セクション

### 2.1 CRC-32C と GF(2) 線形性

CRC-32C は GF(2) 上の **アフィン写像**:

```
crc(M) = c_L ⊕ f(M)         (L = len(M))
```

ここで `f` は GF(2) 線形:

```
f(M1 ⊕ M2) = f(M1) ⊕ f(M2)
```

そして `c_L = crc(0_L)` (長さ L の全ゼロメッセージの CRC)。実装上は init `0xFFFF_FFFF` と final XOR `0xFFFF_FFFF` がアフィン定数 `c_L` に集約される。

#### 線形性が許す/禁ずるもの

- **許す**: `popcount-XOR` 方式 (各メッセージビット `i` に対応する事前計算済み 32 ビット定数 `G_L[i]` の XOR 累積)。1024 ビットフレームに対し 1024 個の 32-bit 定数 = 4 KiB のテーブルで O(L) 処理が可能
- **許す**: SIMD レーンに「同じ長さの異なるメッセージを並べて並列処理する」 (ただし bit-by-bit 部分が直列なので、レーン数倍のスループットしか得られない)
- **禁ずる**: 1 メッセージを SIMD で並列分割して処理すること (CRC は本質的に message ordering に依存する直列演算; SIMD で速くするには PCLMULQDQ などの **carryless multiply** 命令が必須で、SSE2/AVX2 単体では直接表現できない)

### 2.2 reflected ポリ表現

CRC-32C の標準実装はすべて **reflected** (LSB-first) 形式を使う:

- normal poly: `0x1EDC_6F41`
- reflected poly: `0x82F6_3B78`
- init: `0xFFFF_FFFF`, final XOR: `0xFFFF_FFFF`
- 各バイトは LSB から処理

bit-by-bit branchless 形:

```rust
state ^= byte as u32;
for _ in 0..8 {
    let mask = (state & 1).wrapping_neg();   // 0xFFFFFFFF if bit 0 set, else 0
    state = (state >> 1) ^ (POLY_REFLECTED & mask);
}
```

`mask` を介した条件 XOR で **分岐ゼロ・定数時間** が達成される。サイドチャネル耐性 (timing leak の最小化) 観点では重要。

### 2.3 SIMD 並列化の 2 アプローチ

- **アプローチ A: 1 メッセージ内並列**: PCLMULQDQ (carryless multiply) で 64..512 bit 単位の polynomial multiply を一発で実行。Intel Whitepaper "Fast CRC Computation Using PCLMULQDQ Instruction" の手法。本タスクでは扱わない (PCLMULQDQ は CLMUL 拡張命令で、純粋な SSE2/AVX2 では表現できない)
- **アプローチ B: 複数メッセージ並列**: 4 (SSE2) または 8 (AVX2) の独立メッセージを `__m128i` / `__m256i` の各レーンに packing し、bit-by-bit branchless を全レーン同時実行。本タスクの SIMD 実装はこれを採る

アプローチ B はメッセージ間に独立性があるとき (例: ストリーム処理で複数チャンクを同時受信、バッチ verify) のみ意味があり、単一メッセージ処理では単に bit-by-bit 直列と同じ。

### 2.4 popcount-XOR の理論的上限

長さ L bit のメッセージで、`G_L[i]` (i = 0..L) を「i 番目の単位ベクトルの CRC から定数 c_L を引いたもの」として事前計算すれば:

```
crc(M) = c_L ⊕ ⊕_{i: M_i = 1} G_L[i]
```

メモリアクセス: L 個の 32-bit ロード (4 KiB / 1024 bit; L1 cache に収まる)
演算: L 回の (mask & XOR)、すなわち branchless でも L 命令必要

L = 1024 の場合、理想的には 1024 サイクル ≈ 200..400 ns/frame 程度になるはず。本実験では 620 ns/frame (median) なので、ループ展開や ILP の最適化余地はあるが、**ハード命令版 (40 ns/frame) の 15 倍遅い** という結論は変わらない。

### 2.5 期待される結果

|  | 期待 |
|---|---|
| `crc32c_hw` | 最速 (40..60 ns/frame, SSE 4.2 命令直接) |
| `crc32c_table` | 中速 (250..400 ns/frame, 1 テーブル参照/byte) |
| `crc32c_slice_by_8` | やや遅い〜中速 (80..150 ns/frame, 8 テーブル並列参照) |
| `crc32c_branchless_bitwise(_v2)` | 遅い (500..700 ns/frame, 8 イテレーション/byte の直列) |
| `crc32c_popcount_xor` | 遅い (500..900 ns/frame, L 回の masked XOR) |
| `crc32c_simd_sse2_x4` | per-frame 中速 (200..300 ns/frame after amortise; 4 lane 並列) |
| `crc32c_simd_avx2_x4` | SSE2 と同等または微遅 (上位 4 レーン未使用; AVX2-VEX エンコーディングのオーバーヘッドだけ得) |

## 4. 考察

### 4.1 結論: ハードウェア命令の独走

**`crc32c_hw` (40 ns/frame) は次点 (`crc32c_slice_by_8` の 90 ns/frame) の 2.3 倍速**。テーブル方式や SIMD 並列化はいずれも追いつけない。スループットでは `crc32c_hw` が **2.58 GiB/s** に達し、メモリ帯域 (DDR4 の単一チャネル 25 GB/s 級) に対して 1/10 オーダーで CRC 計算が直線律速にならない領域に入っている。

**production の `update_crc` / `verify_crc` は `crc32c` クレートのまま据え置くべき**、という判断を再確認する数値根拠が得られた。

### 4.2 ランキングと考察

| 順位 | 実装 | median ns/frame | 対 hw 比 | コメント |
|:---:|---|---:|---:|---|
| 1 | `crc32c_hw` | 40 | 1.0× | SSE 4.2 命令、並ぶものなし |
| 2 | `crc32c_slice_by_8` | 90 | 2.3× | 純ソフトウェアでは最速 |
| 3 | `crc32c_table` (Sarwate) | 310 | 7.8× | シンプル、メモリ局所性も高い |
| 4 | `crc32c_simd_sse2_x4` | 790/4 ≈ 198 | 5.0× (per-frame) | 4 メッセージ同時処理時のみ意味あり |
| 5 | `crc32c_simd_avx2_x4` | 1040/4 ≈ 260 | 6.5× (per-frame) | **SSE2 より遅い** (4.3 節参照) |
| 6 | `crc32c_branchless_bitwise_v2` | 560 | 14.0× | xnor/nand/xor3 経由でも v1 と差なし |
| 7 | `crc32c_branchless_bitwise` | 620 | 15.5× | サイドチャネル耐性目的の参照実装 |
| 8 | `crc32c_popcount_xor` | 620 | 15.5× | 線形性活用、テーブルアクセス支配 |

### 4.3 AVX2 が SSE2 より遅い理由 (重要な発見)

直観に反して `crc32c_simd_avx2_x4` (1040 ns/4-frame) は `crc32c_simd_sse2_x4` (790 ns/4-frame) **より遅い**。原因は本実装が **AVX2 の上位 4 レーン (4..7) を意図的に未使用** にしているため:

- AVX2 命令は 256 bit 全体を処理する命令幅で、下位 128 bit のみ使う場合でも実行ユニットの占有時間は同じ
- AVX2 命令 (VEX エンコーディング) は SSE2 命令より 1 byte 長く、命令キャッシュ効率がわずかに劣る
- レーン数 4 で固定するなら SSE2 命令で十分、AVX2 を使う意味はない

**設計判断**: AVX2 を活かすには `crc32c_simd_avx2_x8(messages: [&[u8]; 8]) -> [u32; 8]` として 8 メッセージ並列にすべき。本実験ではタスクスペックに合わせて `_x4` 版を実装したが、production 用途では `_x8` 版が適切 (フォローアップ候補)。

### 4.4 popcount-XOR の限界

`crc32c_popcount_xor` (620 ns/frame) は Sarwate 8-bit (310 ns/frame) より遅い。原因:

- L = 1024 個の 32-bit basis をすべて XOR 累積する必要があり、ベース演算量が table 方式 (128 回のテーブル参照) の 8 倍
- branchless mask (`lsb_mask(b >> bit_i) & basis[i]`) は分岐回避のために常に XOR 命令を発行 (table 方式は分岐回避を必要としない)
- L1 キャッシュアクセスは両方 (4 KiB basis vs 1 KiB Sarwate table) でも同等

**結論**: popcount-XOR は理論的には美しい (CRC の線形性を完全に表現) が、**実用的には Sarwate テーブル法に勝てない**。サイドチャネル耐性が真に必要な暗号文脈なら採用検討の価値はある (アクセスパターンが完全に決定論的)。

### 4.5 branchless bitwise 二版の差

`v1` (620 ns) と `v2` (xnor/nand/xor3 経由, 560 ns) で **約 10% の差** がある。理由は機械的: `v2` は中間値が `xor3(state >> 1, !nand(POLY, mask), 0)` のように **三項 XOR** に展開される箇所があり、LLVM バックエンドで `XOR + XOR` 2 命令に分解される。`v1` は `state = (state >> 1) ^ (POLY & mask)` で 2 命令直接。

10% 差は誤差範囲内とも言える (実行ごとに ±20 ns 程度の variance がある)。両方とも **同じ計算をするので bit-exact 一致**。

### 4.6 サイドチャネル耐性との関係

本実験は「分岐ゼロ実装」をベンチで動作確認したが、**形式検証は別タスク**。`branchless_bitwise{,v2}` および `popcount_xor` は分岐なし・データ依存メモリアクセスなしで実装されており、**timing side-channel の主要 vector を排除する候補** として位置付けられる。一方 `crc32c_table` / `crc32c_slice_by_8` は **テーブル参照のキャッシュタイミングがリーク源**になり得るため、暗号文脈では避けるべき。

### 4.7 何を確定できたか / できなかったか

- ✅ **production の `crc32c` クレート (= `crc32c_hw`) を維持すべき**。代替実装はいずれも 2.3 倍以上遅く、機能的優位 (移植性、サイドチャネル耐性) のために性能を捨てる場合のみ意味あり
- ✅ **8 つの実装すべてが bit-exact 一致** (`tests/crc_impls_consistency.rs` の 10000 random message × 4 lane = 40000 sample で確認)
- ✅ **AVX2 で SSE2 より遅くなる場合がある** という反直観的な発見。レーン数を変えずに命令幅だけ広げても無意味
- ✅ **popcount-XOR 実装は線形性の "理論的美しさ" を持つが、実装としては Sarwate テーブル法に劣る**
- ❌ **PCLMULQDQ 実装は本実験範囲外**。1 メッセージ内並列で SSE 4.2 を超える (もしくは互角の) 純ソフト実装ができるかは別タスクで検証すべき (フォローアップ候補)
- ❌ **AVX-512 / NEON / SVE は CI ランナーがサポートしていない / クロスコンパイル必要のため別タスク**
- ❌ **形式検証によるサイドチャネル耐性証明は別タスク**

### 4.8 実験の意義

本タスクは設計判断として「`crc32c` クレートを置き換えるべきか?」に対する **明確な "No"** を残した。今後 production 性能改善 / サイドチャネル耐性 / 移植性の観点で代替実装提案が出た場合、本ドキュメントを根拠に「性能で 2 倍以上遅くなる、それを正当化する非性能要因が必須」と要求できる。

**8 実装が bit-exact 一致する性質は将来 PCLMULQDQ / NEON / 暗号文脈実装を追加した際の差分テストの基盤としても再利用可能**。

## 7. Spec からの軽微な逸脱

タスクスペックに対する以下の軽微な変更を記録:

- `crc32c_simd_sse2(data: &[u8]) -> u32` の signature → **`crc32c_simd_sse2_x4(messages: [&[u8]; 4]) -> [u32; 4]`** に変更。元の signature は記述 "4 メッセージ同時 CRC" と矛盾しており、SIMD 並列化の意味が立たない。spec の **意図** (4 メッセージ並列) に従った

- AVX-512 実装は CI ランナーが対応していない可能性が高いため未実装 (spec の "Out of scope" セクションでも明示)
